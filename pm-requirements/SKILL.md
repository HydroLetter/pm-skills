# 产品经理需求分析

本skill覆盖需求分析全流程，包括需求挖掘、分析模型、评估排序、文档输出。

## 流程概览：需求分析标准化流程（8步）

```
用户访谈 → 需求收集 → 需求分类 → 需求建模 → 优先级排序 → 产品方案 → 需求评审 → 输出MRD/PRD
```

| 步骤 | 关键动作 | 产出 | 工具/方法 |
|------|----------|------|-----------|
| 1. 用户访谈 | 目标用户深度访谈，了解真实场景 | 访谈记录 | 用户画像、场景分析 |
| 2. 需求收集 | 多渠道收集需求（调研/竞品/数据/反馈） | 需求池 | 用户反馈系统、竞品分析 |
| 3. 需求分类 | 按维度归类：功能需求/体验需求/商业需求 | 分类需求清单 | 需求分类矩阵 |
| 4. 需求建模 | 用分析模型工具结构化梳理 | 需求模型图 | KANO/CES/用户故事 |
| 5. 优先级排序 | 多维度打分排序 | 优先级排序表 | RICE/MoSCoW/价值-成本矩阵 |
| 6. 产品方案 | 将需求转化为产品方案 | 方案文档 | 业务流程图/原型 |
| 7. 需求评审 | 多方评审，对齐需求理解 | 评审结论 | 评审会/邮件评审 |
| 8. 输出文档 | 最终输出需求文档 | PRD/MRD | 标准化文档模板 |

## 核心模型一：产品需求分析模型

### KANO模型（需求价值分类）

| 需求类型 | 定义 | 特征 | 产品策略 |
|----------|------|------|----------|
| **基本型需求** | 不做用户会不满意 | 用户不会主动提，但缺失必投诉 | 必须满足，做好是应该的 |
| **期望型需求** | 做得越好用户越满意 | 用户会明确提出，竞品对比重点 | 持续优化，建立优势 |
| **兴奋型需求** | 做了超出用户预期 | 用户自己都没想到 | 少量投入制造亮点 |
| **无差异需求** | 做不做用户没感觉 | ROI极低 | 直接拒绝 |
| **反向需求** | 做了反而引起不满 | 违背用户习惯 | 坚决避免 |

### 用户场景分析法

```
用户是谁 → 在什么场景下 → 想完成什么目标 → 遇到了什么困难 → 现有的解决方案是什么 → 我们能不能做得更好
```

### 需求优先级模型（价值-成本矩阵）

```
        高价值
          ↑
  优先投入  |  战略性投入
  (Quick Win) | (长期规划)
  ——————————+——————————→ 高成本
  选择性做  |  不做
  (维持)    | (拒绝)
          ↓ 低价值
```

## 核心模型二：产品逻辑确定方法

### 三层分析框架

```
第一层：用户价值
  ├→ 用户痛点是什么？（用数据说话）
  ├→ 用户当前如何解决？（替代方案分析）
  └→ 我们的方案好在哪里？（差异化价值）

第二层：商业价值
  ├→ 市场规模和增长潜力
  ├→ 付费意愿验证
  └→ 商业模型可行性

第三层：技术可行性
  ├→ 技术难度评估
  ├→ 开发资源需求
  └→ 与现有系统兼容性
```

### 产品逻辑检验清单

| 检验维度 | 核心问题 | 通过标准 |
|----------|----------|----------|
| 需求真伪 | 用户真的有这个需求吗？ | 有用户访谈/数据支撑 |
| 需求强度 | 用户有多痛？不解决会怎样？ | ≥3个独立来源交叉验证 |
| 市场空间 | 这个需求覆盖多少用户？ | TAM/SAM/SOM可量化 |
| 商业闭环 | 用户愿意付费吗？怎么赚钱？ | 有付费验证或类似案例 |
| 竞争优势 | 为什么是我们不是别人？ | 有明确的差异化壁垒 |

## 核心模型三：需求分析流程 — 从需求到功能

```
需求池入口 → 需求评审 → 可行性分析 →
  版本规划 → 功能设计 → 交互设计 →
    评审确认 → 开发排期 → 跟踪验收
```

### 需求池管理规范

| 字段 | 说明 | 示例 |
|------|------|------|
| 需求来源 | 谁提出的 | 用户反馈/运营要求/老板需求 |
| 需求描述 | 一句话描述 | 用户可以导出订单记录 |
| 需求类型 | 功能/优化/BUG | 功能新增 |
| 优先级 | P0/P1/P2/P3 | P1 |
| 所属版本 | 计划在哪个版本做 | V2.3 |
| 状态 | 待评审/已确认/开发中/已上线 | 待评审 |

## 专项：算法产品需求分析流程

> 算法/AI产品与常规功能产品的需求分析有本质区别

```
业务目标明确 → 数据调研（是否有可用数据/数据质量如何）
        ↓
算法方案设计 → 评估指标定义（准确率/召回率/覆盖率等）
        ↓
数据标注需求 → 模型训练 → 效果验证 → 实验上线
        ↓
业务反馈 → 持续优化迭代
```

| 阶段 | 常规产品PM | 算法产品PM额外关注 |
|------|-----------|-------------------|
| 需求调研 | 用户痛点 | 数据可得性、标注成本 |
| 方案设计 | 功能流程 | 算法策略、评估指标 |
| 开发阶段 | 开发跟进 | 数据标注进度、训练效果 |
| 验收阶段 | 功能验收 | 离线指标+在线A/B实验 |
| 上线后 | 用户反馈 | Badcase分析、模型迭代 |

**算法需求分析检查项**：
- [ ] 业务指标是否可转化为可优化的算法指标？
- [ ] 是否有足够的训练数据？数据质量如何？
- [ ] 数据标注成本是否在预算范围内？
- [ ] 模型效果的好坏如何量化衡量？
- [ ] 上线后如何持续收集反馈优化模型？

## 需求分析常见误区

| 误区 | 正确做法 |
|------|----------|
| 用户说什么就做什么 | 区分"用户说的"和"用户真正需要的" |
| 功能堆砌 | 宁少勿多，用最小功能集验证价值 |
| 忽视场景 | 同一需求在不同场景下含义不同 |
| 不做竞品调研 | 先看竞品怎么做，避免重新发明轮子 |
| 技术驱动 | 需求决定技术，不是技术决定需求 |
| 没有数据支撑 | 每个需求尽量找到数据证据 |

## 常见问题速查

**Q: 用户需求和产品需求有什么区别？**
A: 用户需求=用户在特定场景下想达成的目标（如：我想快速找到想要的衣服）；产品需求=PM设计的解决方案（如：智能搜索+筛选+推荐）

**Q: KANO模型怎么用？**
A: 通过用户问卷（正反两问）判断需求属于哪种类型：做这个功能你会满意吗？不做这个功能你会不满吗？交叉分析得出需求类型。

**Q: 需求优先级怎么排？**
A: 建议使用RICE：Reach(覆盖人数) × Impact(影响程度) × Confidence(信心) / Effort(工作量)。同时考虑战略价值和依赖关系。

**Q: 需求文档写多细？**
A: 核心要素必须清晰：背景+目标+用户场景+功能描述+交互规则+异常处理+验收标准。细节程度取决于团队协作方式（重文档 vs 轻文档+多沟通）。

**Q: 算法产品的需求分析有什么不同？**
A: 核心差异：1)需要定义算法评估指标 2)需要考虑数据可用性 3)效果有不确定性需要A/B实验验证 4)迭代节奏不同于功能开发。
