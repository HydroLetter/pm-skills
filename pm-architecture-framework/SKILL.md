# 产品架构设计框架与平台架构

本skill提供产品架构设计的方法论、平台架构模式和基础设施架构指南。

## 一、产品架构设计方法论

### 什么是产品架构

产品架构是产品的骨架，定义了：
- **功能模块划分**：产品由哪些功能模块组成
- **模块间关系**：模块之间如何协作、依赖
- **数据流转**：数据如何在模块间流动
- **技术边界**：哪些功能自建、哪些对接第三方

### 产品架构设计四步法

```
1. 业务分析 → 梳理业务流程、识别核心场景
        ↓
2. 功能拆解 → 按业务域拆分功能模块
        ↓
3. 架构设计 → 确定分层、定义模块边界
        ↓
4. 接口定义 → 模块间交互协议、数据格式
```

### 产品功能矩阵设计

功能矩阵是产品架构的二维表达：

| 维度 | 说明 | 示例 |
|------|------|------|
| **用户维度** | 不同用户角色的功能需求 | B端/C端/管理员 |
| **场景维度** | 不同业务场景的功能组合 | 购买/售后/营销 |
| **层级维度** | 功能的层级关系 | 前台/中台/后台 |

**设计原则**：
- 同一层级的功能相互独立、完全穷尽(MECE)
- 上层功能可调用下层能力，避免跨层调用
- 核心功能下沉，通用能力抽象

## 二、分层架构模式

### 经典四层架构

```
┌─────────────────────────────────────┐
│  表现层(用户界面)                    │  Web/App/小程序/管理后台
├─────────────────────────────────────┤
│  应用层(业务逻辑)                    │  订单/支付/用户/商品服务
├─────────────────────────────────────┤
│  服务层(通用能力)                    │  消息/权限/日志/搜索服务
├─────────────────────────────────────┤
│  数据层(存储持久化)                  │  MySQL/Redis/MongoDB
└─────────────────────────────────────┘
```

### 微服务架构

**服务拆分原则**：
| 原则 | 说明 | 示例 |
|------|------|------|
| 按业务域拆分 | 高内聚低耦合 | 用户服务/订单服务/支付服务 |
| 按读写拆分 | CQRS模式 | 读服务/写服务分离 |
| 按优先级拆分 | 核心/非核心隔离 | 核心交易/营销活动 |

**服务间通信**：
- 同步调用：RESTful API、gRPC
- 异步消息：Kafka、RabbitMQ
- 事件驱动：Event Bus

## 三、中台架构

### 业务中台

业务中台将通用业务能力抽象复用：

| 中台模块 | 核心能力 | 服务对象 |
|----------|----------|----------|
| **用户中心** | 注册/登录/权限/画像 | 全业务线 |
| **商品中心** | 类目/属性/库存/价格 | 电商业务 |
| **交易中心** | 订单/购物车/履约 | 交易业务 |
| **支付中心** | 多渠道支付/对账/退款 | 全业务线 |
| **营销中心** | 优惠券/满减/积分 | 运营活动 |

### 数据中台

数据中台架构：

```
数据采集 → 数据存储 → 数据治理 → 数据服务 → 数据应用
   ↓           ↓           ↓           ↓           ↓
 日志/业务   数据湖/仓库   清洗/建模   API/报表   BI/AI
```

**核心组件**：
| 组件 | 功能 | 技术选型 |
|------|------|----------|
| 数据采集 | 多源数据接入 | Flume/Kafka |
| 数据存储 | 海量数据存储 | HDFS/HBase |
| 数据计算 | 离线/实时计算 | Spark/Flink |
| 数据服务 | API化数据能力 | Data API |

### AI中台

AI中台架构分层：

| 层级 | 能力 | 说明 |
|------|------|------|
| **应用层** | 智能客服/推荐/风控 | 业务场景落地 |
| **能力层** | 文字/图像/语音/多模态生成 | 通用AI能力 |
| **技术层** | NLP/CV/多模态大模型 | 算法模型 |
| **基础设施** | 算力/数据/开发平台 | 模型训练支撑 |

**AI中台-算法层**：
- 图像AI：人脸识别、物体检测、OCR
- 文本AI：分词、语义分析、NER、文档翻译
- 语音AI：语音识别、合成、分析

## 四、企业级架构

### ESB企业服务总线

ESB核心功能：
- **协议转换**：HTTP/HTTPS/SOAP/MQ
- **路由转发**：根据规则路由请求
- **服务编排**：组合多个服务
- **监控治理**：服务调用链追踪

**适用场景**：
- 企业内异构系统集成
- 遗留系统现代化改造
- 统一服务接口规范

### 企业IT系统架构

典型企业IT系统分层：

```
决策层：BI/EPM/绩效管理
    ↓
运营层：ERP/CRM/SCM/PLM
    ↓
支持层：HR/财务/OA/ITSM
    ↓
执行层：MES/WMS/TMS/APS
```

**核心系统说明**：
| 系统 | 全称 | 核心功能 |
|------|------|----------|
| ERP | 企业资源计划 | 财务/供应链/生产 |
| CRM | 客户关系管理 | 销售/市场/服务 |
| SCM | 供应链管理 | 采购/库存/物流 |
| MES | 制造执行系统 | 生产调度/质检 |
| WMS | 仓库管理系统 | 入库/出库/盘点 |
| PLM | 产品生命周期 | 研发/设计/文档 |

## 七、架构设计检查清单

### 设计前检查

- [ ] 业务目标和范围是否清晰？
- [ ] 核心用户场景是否已识别？
- [ ] 功能边界是否明确（做/不做）？
- [ ] 非功能性需求是否明确（性能/安全/扩展性）？

### 设计中检查

- [ ] 模块划分是否符合高内聚低耦合？
- [ ] 数据流向是否清晰？
- [ ] 关键路径是否有单点故障？
- [ ] 扩展性是否预留空间？

### 设计后检查

- [ ] 技术选型是否有团队支撑？
- [ ] 第三方依赖是否可控？
- [ ] 监控告警是否覆盖？
- [ ] 回滚方案是否就绪？

## 八、架构图绘制规范

### 架构图类型选择

| 类型 | 用途 | 受众 |
|------|------|------|
| **业务架构图** | 展示业务模块关系 | 业务方/管理层 |
| **应用架构图** | 展示应用系统划分 | 产品经理/架构师 |
| **技术架构图** | 展示技术组件关系 | 技术团队 |
| **部署架构图** | 展示物理部署拓扑 | 运维团队 |

### 架构图绘制原则

1. **分层清晰**：从上到下或从左到右有明确流向
2. **边界明确**：模块边界清晰，命名规范
3. **重点突出**：核心模块突出显示
4. **图例完整**：必要的图例说明
