# 产品经理策略工具箱

本skill整合PM日常工作中最常用的策略工具、商业决策方法和操作指南。

## 一、产品定价逻辑与流程

### 定价六步法

```
1. 明确定价目标 → 2. 分析成本结构 → 3. 研究竞品定价 → 4. 评估用户支付意愿
                                    ↓
                    6. 持续调优 ← 5. 选择定价模型
```

### 三种核心定价模型

| 模型 | 逻辑 | 适用场景 | 优势 | 劣势 |
|------|------|----------|------|------|
| **成本导向** | 成本 + 预期利润 = 价格 | 制造业、硬件 | 简单清晰 | 忽视市场 |
| **竞争导向** | 对标竞品定价 | 成熟市场、标品 | 市场验证过 | 差异化难体现 |
| **价值导向** | 用户感知价值 = 价格 | 创新产品、SaaS | 利润空间大 | 量化价值难 |

### 常见定价策略矩阵

```
        渗透定价(低) ── 快速获客，牺牲短期利润
               ↓
  价格 ←── 市场定价 ── 对标竞品
               ↓
        溢价定价(高) ── 品牌/价值支撑高价格
              ← →
        免费增值(Freemium) ── 基础免费 + 高级付费
        订阅制 ── 按月/年收费
        按量付费 ── 用得越多付得越多
        一次性买断 ── 永久使用
```

### 定价验证清单

- [ ] 目标用户群体的支付意愿是否验证过？
- [ ] 定价能否覆盖获客成本+运维成本？
- [ ] 竞品定价是多少？用户为什么选我们？
- [ ] 价格弹性测试过吗（涨价/降价对用户数的影响）？
- [ ] 有没有设计价格锚点（高价版本衬托主推版本）？

## 二、竞品分析六步法

```
第一步：明确分析目标 → 为什么要做竞品分析？
第二步：圈定竞品范围 → 直接竞品 / 间接竞品 / 替代品
第三步：信息收集 → 产品体验 + 公开信息 + 用户评价
第四步：对比分析 → 功能/体验/定价/用户评价/商业模式
第五步：洞察输出 → SWOT结论 + 差异化策略
第六步：持续追踪 → 竞品追踪表
```

### 竞品分析内容框架

| 维度 | 分析内容 | 信息来源 |
|------|----------|----------|
| **产品功能** | 核心功能列表、差异化功能、迭代速度 | 产品体验、App更新日志 |
| **用户体验** | 交互设计、视觉风格、用户评价 | 亲自体验、应用商店评价 |
| **定价策略** | 定价模型、价格区间、优惠策略 | 官网、客服咨询 |
| **商业模式** | 如何赚钱、盈利效率、客户LTV | 财报、分析报告 |
| **市场表现** | 用户规模、市场份额、增长趋势 | 第三方数据、招聘信息 |
| **技术能力** | 技术架构、专利、团队背景 | 技术博客、招聘JD |

### 竞品分析输出模板

```markdown
# XX产品竞品分析报告
## 1. 分析目标
## 2. 竞品概览（基本信息对比表）
## 3. 功能对比矩阵
## 4. 优劣势分析（SWOT）
## 5. 差异化策略建议
## 6. 行动建议（短期/中期/长期）
```

## 三、产品工作全流程（PM日常工作视角）

### PM完整工作流

```
发现问题 → 需求定义 → 产品规划 → 产品设计 → 项目管理 → 数据分析 → 反馈迭代
（持续循环）
```

### PM工作周期（以周为单位）

| 日 | 核心工作 | 产出 |
|----|----------|------|
| 周一 | 周会，明确本周优先级 | 本周计划 |
| 周二 | 深度产品设计（PRD/原型） | 设计方案 |
| 周三 | 开发进度跟进、需求答疑 | 进度更新 |
| 周四 | 数据分析、用户调研 | 数据报告 |
| 周五 | 周报、复盘、下周准备 | 周报 |
| 周末 | 竞品调研、行业学习 | 行业洞察 |

### PM核心工作内容

1. **需求管理**（30%）：需求收集 → 分析 → 排序 → 文档输出 → 评审讲需求
2. **产品设计**（25%）：流程图 → 原型图 → PRD → 评审方案
3. **项目管理**（20%）：进度跟踪 → 风险识别 → 资源协调 → 上线验收
4. **数据分析**（15%）：埋点设计 → 数据看板 → 效果评估 → 决策支持
5. **行业学习**（10%）：竞品体验 → 前沿技术 → 用户洞察

## 四、产品经理自检清单

### 需求阶段自检

- [ ] 这个需求解决的是真问题还是伪需求？（有用户证据支持吗？）
- [ ] 不做这个需求，用户有什么替代方案？
- [ ] 这个需求的ROI够高吗？（投入 vs 产出）
- [ ] 是否找到了最小可行方案？（MVP思路）
- [ ] 异常情况都考虑到了吗？

### 设计阶段自检

- [ ] 流程是否完整？有没有遗漏的步骤？
- [ ] 核心路径是否最简？用户需要几步完成目标？
- [ ] 极端情况有处置方案吗？（空页面、断网、超时）
- [ ] 信息层级是否清晰？用户一眼能看到最重要的吗？

### 开发阶段自检

- [ ] PRD是否足够清晰？研发有无理解偏差？
- [ ] 需求变更是否控制在合理范围内？
- [ ] 高风险（核心路径/支付相关）是否有额外的质量保障？
- [ ] 每日进度是否在计划范围内？

### 上线前自检

- [ ] 所有核心功能是否验收通过？
- [ ] BUG修复情况如何？（无Blocker/Critical）
- [ ] 数据埋点是否部署并验证？
- [ ] 运营/客服是否已培训？
- [ ] 回滚方案是否就绪？

### 上线后自检

- [ ] 核心指标是否正常？（无异常下降）
- [ ] 用户反馈渠道是否通畅？
- [ ] 是否有线上BUG需要紧急修复？
- [ ] 本次需求的效果是否可以量化？
- [ ] 下次迭代的方向是否已识别？

## 五、数据分析报表方法论

### 数据分析四步法

```
第一步：明确分析目的 → 要回答什么问题？
第二步：数据准备 → 需要什么数据？数据质量如何？埋点是否到位？
第三步：分析执行 → 趋势/对比/分组/归因
第四步：结论输出 → 做了什么？为什么？下一步怎么做？
```

### 报表撰写结构

```
1. 核心结论（一句话）
2. 关键数据概览（趋势图+核心指标）
3. 详细分析（分维度/分渠道）
4. 问题诊断（数据异常的原因分析）
5. 行动建议（短期/中期/长期）
```

### PM常用数据分析维度

| 分析类型 | 常用指标 | 目的 |
|----------|----------|------|
| 用户增长 | DAU/MAU、新增/留存/回流、获客成本 | 判断增长健康度 |
| 用户行为 | PV/UV、点击率、停留时长、转化率 | 判断产品体验 |
| 商业效果 | GMV、付费率、ARPU、LTV | 判断商业健康度 |
| 产品质量 | 崩溃率、加载时长、BUG率 | 判断产品稳定性 |

### 数据分析常见陷阱

| 陷阱 | 说明 |
|------|------|
| 幸存者偏差 | 只看到成功案例，忽略沉默的大多数 |
| 相关性≠因果性 | 两个指标同步变化不一定是因果关系 |
| 数据口径不一致 | 不同平台/VERSION的数据口径要统一 |
| 只看平均水平 | 均值可能掩盖极端值，关注分布 |
| 过度解读波动 | 短期波动不一定是趋势，需要持续观察 |

## 常见问题速查

**Q: 怎么给新产品定价？**
A: 1)先算清楚成本底线 2)调研竞品定价 3)小范围测试用户对价格的反应 4)从低到高调整比从高到低容易。

**Q: 竞品分析到什么程度算够？**
A: 核心功能体验完+了解定价+知道优劣势+能输出差异化策略。不需要穷举所有细节。

**Q: PM日常被各种琐事打断怎么管理时间？**
A: 1)上午做大块深度工作（PRD/原型） 2)下午处理沟通和会议 3)紧急事才立即响应 4)周计划比日计划更实用。

**Q: 数据分析发现不了问题怎么办？**
A: 1)换维度看（按用户类型/渠道/时间分段） 2)对比看（同比/环比/竞品对比） 3)从异常值入手（最高/最低/变化最大的） 4)带着假设去验证而非漫无目的看数据。
