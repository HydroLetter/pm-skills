# 产品设计与研发流程

本skill覆盖产品从设计阶段到研发交付的全流程方法论。

## 一、产品设计流程

### 设计阶段全流程

```
需求确认 → 设计启动 → 方案设计 → 原型输出 → 设计评审 → 设计定稿 → 交付研发
```

| 设计阶段 | 产品经理 | UI/UX设计 | 关键产出 |
|----------|----------|-----------|----------|
| 需求确认 | 确保需求已评审通过 | — | 已确认PRD |
| 设计启动 | 明确设计目标、提供业务流程图 | 理解需求、确认设计排期 | 设计任务单 |
| 方案设计 | 参与思路讨论、给出建议 | 低保真线框图 → 交互方案 | 线框图/交互稿 |
| 原型输出 | 确认交互逻辑和业务规则 | 高保真原型 → 视觉设计 | 高保真原型 |
| 设计评审 | 主持评审、收集意见、评估可行性 | 讲解设计思路、收集反馈 | 评审纪要 |
| 设计定稿 | 签字确认、控制变更 | 输出最终设计稿+标注+切图 | 设计终稿 |
| 交付研发 | 提供业务规则说明、解答设计问题 | 提供设计资源包、建立设计规范 | 设计规范文档 |

### 设计决策原则

- **先结构后表现**：先确定信息架构和流程，再讨论视觉
- **先易用后好看**：可用性 > 美观度
- **数据驱动**：设计决策要有用户数据支撑，不要凭个人审美
- **移动端优先**：先设计移动端，再适配PC端

## 二、产品研发流程

### 研发阶段完整流程

```
PRD输出 → 技术评审 → 技术方案设计 → 编码开发
    ↓          ↓
排期确认 ← 工作量评估
    ↓
编码实现（前端+后端并行）→ 联调 → 自测 → 提测
    ↓
测试阶段 → BUG修复 → 回归验证 → 验收 → 发版上线
```

### 研发各角色协作

```
产品经理 ──→ 提供PRD/原型/业务规则 ──→ 解答开发过程中的业务问题
   ↕                ↕
研发小组 ──→ 技术评审反馈 ──→ 技术方案 ──→ 排期 ──→ 编码 ──→ 提测
   ↕                ↕
测试团队 ──→ 测试用例 ──→ 功能测试 ──→ 兼容测试 ──→ 回归测试 ──→ 输出测试报告
```

### PM在研发阶段的核心职责

1. **需求解释者**：确保研发正确理解需求，不要假设"研发肯定懂"
2. **变更管控者**：控制需求变更，评估变更影响
3. **进度跟踪者**：每日了解开发进度，及时发现风险
4. **决策加速者**：开发过程中遇到业务逻辑不确定时快速拍板
5. **验收把关者**：对照PRD逐条验收，不符合的坚决退回

## 三、产品经理技术知识体系

> PM不需要会写代码，但需要理解技术边界和协作语言

### 必懂概念框架

```
前端（用户看到的）          后端（用户看不到的）         数据/存储
    ├→ HTML/CSS/JavaScript     ├→ 接口(API)设计            ├→ 数据库(MySQL/Redis)
    ├→ 响应式布局              ├→ 服务端逻辑                ├→ 数据存储结构
    ├→ 页面跳转逻辑            ├→ 用户认证/权限             ├→ 缓存策略
    │                          ├→ 消息推送                  │
    │                          └→ 定时任务/后台管理          └→ 数据埋点/上报
```

### 技术沟通关键词

| 概念 | PM需要知道的 | 为什么重要 |
|------|-------------|-----------|
| **接口(API)** | 前后端通过接口传递数据；接口文档定义"请求什么→返回什么" | 一个功能是否涉及前后端联调，影响排期 |
| **数据库** | 数据存在哪里、怎么存，影响数据查询和分析 | 影响功能设计（如搜索、排序、报表） |
| **缓存** | 常用数据存缓存不查数据库，速度快 | 影响性能体验和数据实时性 |
| **版本控制** | Git/SVN记录代码变更历史 | 理解发版流程和回滚机制 |
| **环境** | 开发/测试/预发布/生产 四个环境 | 了解代码从开发到上线的路径 |
| **架构** | 系统的大结构设计 | 大功能修改是否影响其他模块 |

### PM的技术能力要求

| 级别 | 要求 |
|------|------|
| 基础 | 能看懂接口文档、理解前后端分工、知道一个功能涉及哪些技术 |
| 进阶 | 能评估技术工作量粗略大小、理解技术方案中的关键决策点 |
| 高级 | 能参与技术方案讨论、能预判技术方案的长期影响 |

## 四、敏捷开发流程

### Scrum开发框架

```
Product Backlog → Sprint Planning → Sprint(2-4周) → Daily Standup
       ↑              ↓                  ↓
       └──── Sprint Review/Retrospective ←─ 可交付增量
```

### Sprint周期内的核心仪式

| 仪式 | 时长 | 参与人 | 目标 |
|------|------|--------|------|
| Sprint Planning | 2-4小时 | 全员 | 确定Sprint目标，拆解任务 |
| Daily Standup | 15分钟 | 全员 | 昨天做了什么？今天做什么？遇到什么阻碍？ |
| Sprint Review | 1-2小时 | 全员+stakeholder | 演示已完成功能，收集反馈 |
| Sprint Retro | 1-1.5小时 | 全员 | 复盘过程，持续改进 |

### Product Backlog管理

| 元素 | 说明 |
|------|------|
| User Story | 作为<角色>，我希望<功能>，以便<价值> |
| Acceptance Criteria | 验收条件，用什么标准判断完成 |
| Story Points | 相对工作量估算，不直接用"人天" |
| Priority | 每次Sprint Planning前重新排序 |

### 敏捷 vs 瀑布

| 维度 | 敏捷 | 瀑布 |
|------|------|------|
| 需求 | 拥抱变化 | 需求固定 |
| 交付 | 持续交付小增量 | 一次性交付完整产品 |
| 反馈 | 每Sprint获取反馈 | 项目结束后获取反馈 |
| 适用 | 需求不明确的创新产品 | 需求明确的大型项目 |

## 五、产品质量管理流程

### 质量管控模型

```
设计质量 → 评审机制保证
    ↓
代码质量 → Code Review + 编码规范
    ↓
测试质量 → 用例评审 + 覆盖率
    ↓
产品质量 → 线上监控 + 用户反馈闭环
```

### 各阶段质量保障措施

| 阶段 | 质量保障措施 | 负责人 |
|------|-------------|--------|
| 需求阶段 | PRD评审、需求澄清会 | PM |
| 设计阶段 | 设计评审、交互走查 | PM+UI |
| 开发阶段 | Code Review、单元测试、自测 | 研发 |
| 测试阶段 | 用例评审、功能测试、性能测试、兼容测试 | 测试 |
| 验收阶段 | 需求验收、UAT测试 | PM+运营 |
| 上线后 | 线上监控、用户反馈、数据对比 | PM+运维 |

### BUG严重等级定义

| 等级 | 定义 | 举例 | 处理 |
|------|------|------|------|
| Blocker | 核心功能无法使用 | 无法登录、支付失败 | 阻塞发版 |
| Critical | 主要功能受损 | 数据计算错误 | 必须修复 |
| Major | 功能可用但有缺陷 | 页面跳转错误 | 尽量修复 |
| Minor | 体验问题 | 文案错误、对齐问题 | 可排入后续 |
| Trivial | 不影响使用 | 颜色偏好建议 | 择机处理 |

## 常见问题速查

**Q: PM需要会写代码吗？**
A: 不需要会写，但需要理解技术边界。知道什么能做、什么难做、大概需要多少时间。会写SQL查数据是加分项。

**Q: 怎么和技术团队高效沟通？**
A: 1)用PRD+原型降低理解成本 2)用数据而非个人感受做决策 3)尊重技术判断，不要在开发面前"教他们写代码" 4)遇到技术障碍时主动帮忙排除依赖。

**Q: 敏捷和瀑布怎么选？**
A: 需求不确定、团队小、需要快速迭代的创新项目 → 敏捷。需求明确、规模大、有固定交付日期的项目 → 瀑布或混合模式。

**Q: 设计评审被驳回怎么办？**
A: 1)先理解驳回的原因是什么（技术不可行？ROI不够？设计不合理？）2)针对性修改 3)如果分歧严重，拉上决策者一起聊，避免PM和设计僵持。
