# 产品经理项目管理全流程

本skill提供完整的PM项目管理流程框架，涵盖从市场判断到项目上线的全生命周期。

## 流程概览

### 综合流程模型（8阶段）

本skill整合了两个视角的流程模型：

**高级流程（业务视角，6阶段）**：市场判断 → 需求管理 → 需求确认 → 项目执行 → 项目验收 → 项目上线

**执行级流程（产研视角，8阶段）**：调研 → 产研设计 → 需求评审 → 开发 → 测试 → 验收 → 发版 → 总结复盘

| 阶段 | 核心工作 | 主导角色 | 关键产出 |
|------|----------|----------|----------|
| 1. 调研阶段 | 市场机会研究、需求挖掘 | 市场/PM | MRD、调研报告 |
| 2. 产研设计 | 需求设计、原型、UI设计 | PM + UI | 业务流程图、产品原型、设计稿 |
| 3. 需求评审 | 全员评审、确认迭代周期 | PM | 评审结论文档、排期表 |
| 4. 开发阶段 | 设计编码、系统自测、提测 | 研发 | 功能代码、接口文档、提测版本 |
| 5. 测试阶段 | 测试执行、BUG管理、测试报告 | 测试 | 测试报告、BUG清单 |
| 6. 验收阶段 | 需求验收、迭代控制、验收报告 | PM + 验收 | 验收报告、操作手册 |
| 7. 发版阶段 | 发版申请、上线部署、通知 | 研发 + 运维 | 上线版本、配置文档 |
| 8. 总结复盘 | 复盘会议、经验沉淀 | PM | 复盘报告 |

### 核心角色

| 角色 | 主要职责 |
|------|----------|
| **客户代表** | 需求输入、设计评审参与、上线通知 |
| **市场/项目** | 市场机会研究、用户调研、迭代验收配合 |
| **产品经理（PM）** | 需求设计、流程梳理、原型、评审组织、迭代控制、复盘 |
| **UI团队** | 界面设计、切图、设计规范、操作手册辅助 |
| **研发小组** | 技术方案、编码、自测、接口文档、BUG修复、发版 |
| **测试团队** | 用例编写与评审、测试执行、BUG提交与回归、测试报告 |
| **验收团队** | 需求验收、问题反馈、验收报告 |
| **运维团队** | 环境准备、服务监控、实施发版 |

## 详细流程图

### 第一阶段：市场判断（运营经理主导）

```
行业趋势 → 竞品分析 → 商业策略 → 【输出】市场需求文档MRD → 【汇报】给领导汇报方案，预期达成效果
```

| 步骤 | 产出物 | 责任人 |
|------|--------|--------|
| 行业趋势分析 | 趋势报告 | 运营经理 |
| 竞品分析 | 竞品调研报告 | 运营经理 |
| 商业策略 | 商业策略文档 | 运营经理 |
| 输出MRD | 市场需求文档 | 运营经理 |
| 领导汇报 | 汇报方案 | 运营经理 → 产品经理 |

### 第二阶段：需求管理（产品经理主导）

```
内部需求收集 → 用户需求调研 → 需求筛选 → 功能结构梳理 → 功能流程梳理 → 交互原理设计 → 文案整理 → 制作推广海报 → 平面设计排期
            ↑__________________________________________↓
```

| 步骤 | 产出物 | 责任人 | 备注 |
|------|--------|--------|------|
| 内部需求收集 | 需求池 | 产品经理 | 持续收集 |
| 用户需求调研 | 定性定量分析 | 产品经理 | 用户访谈、问卷 |
| 需求筛选 | 版本规划 | 产品经理 | 优先级排序 |
| 功能结构梳理 | 功能结构图 | 产品经理 | 信息架构 |
| 功能流程梳理 | 流程图 | 产品经理 | 业务流程 |
| 交互原理设计 | 交互原型 | 产品经理 | 低保真 |
| 需求确认 | 确认文档 | 产品经理 | 多方对齐 |
| 文案整理校对 | 文案素材 | 文案编辑 | 配合设计 |
| 制作推广海报 | 海报设计稿 | 文案编辑 | 预热准备 |
| 平面设计排期 | 设计排期表 | 文案编辑 | 协调资源 |

### 第三阶段：需求确认（跨角色协作）

```
产品运营推广 → 产品技术讨论 → 产品设计讨论 → 原型评审会 → 项目排期表 → 改版前数据统计 → 产品需求文档中心
   ↓              ↓              ↓              ↓            ↓              ↓
 评审原型    需求功能可行性  排版交互设计  确认原型/排期   确认排期      数据统计基线
```

| 环节 | 参与角色 | 产出物 |
|------|----------|--------|
| 产品运营推广评审原型 | 运营经理 | 原型反馈 |
| 产品技术讨论 | 后端开发 | 技术可行性评估 |
| 产品设计讨论 | 界面设计 | 设计稿/交互方案 |
| 原型评审会 | 全员 | 确认原型、排期 |
| 项目排期表 | 产品经理 | 详细排期 |
| 改版前数据统计 | 运营经理 | 数据基线 |
| 产品需求文档中心 | 产品经理 | PRD文档 |
| 汇报进度 | 产品经理 | 进度汇报给领导 |

### 第四阶段：项目执行（开发主导）

```
【进度】每日下班前群里反馈进度
    ↓
【设计】输出效果图/复杂图切图
    ↓
产品技术运营设计稿评审
    ↓
    ├→ 【前端】重构页面
    └→ 【后端】接口开发
         ↓
    【测试】产品运营测试验收输出BugList
         ↓
    【前端】调试界面BUG
    【后端】后台管理功能开发管理数据
```

| 步骤 | 产出物 | 责任人 |
|------|--------|--------|
| 每日进度反馈 | 进度更新 | 全员 |
| 设计输出 | 效果图、切图 | 界面设计 |
| 设计稿评审 | 评审确认 | 全员 |
| 前端重构 | 页面代码 | 前端开发 |
| 后端开发 | 接口/后台 | 后端开发 |
| 测试验收 | BugList | 运营/产品 |
| BUG修复 | 修复版本 | 前后端 |

### 第五阶段：项目验收

```
通知公司项目参与人员验收
    ↓
收集验收问题，根据紧急程度改版迭代
    ↓
新版上线
    ↓
推广海报定稿
    ↓
渠道合作协调推广
```

| 步骤 | 产出物 | 责任人 |
|------|--------|--------|
| 发起验收通知 | 验收通知 | 产品经理 |
| 收集验收问题 | 问题清单 | 全员 |
| 改版迭代 | 修复版本 | 开发团队 |
| 新版上线 | 上线版本 | 产品经理 |
| 推广海报定稿 | 最终海报 | 运营经理 |
| 渠道推广 | 推广计划 | 运营经理 |

### 第六阶段：项目上线

```
用户回访
    ↓
【反馈】改版前后数据效果对比团队反馈
    ↓
根据数据&回访反馈，规划下次迭代
    ↓
更新需求池版本管理
    ↓
【汇报】给领导汇报结果，及迭代计划
```

| 步骤 | 产出物 | 责任人 |
|------|--------|--------|
| 用户回访 | 用户反馈 | 运营经理 |
| 数据对比 | 效果报告 | 产品经理 |
| 团队反馈 | 反馈汇总 | 产品经理 |
| 迭代规划 | 迭代计划 | 产品经理 |
| 需求池更新 | 更新需求池 | 产品经理 |
| 结果汇报 | 汇报材料 | 产品经理 |

## 关键检查点

### 各阶段必须产出

| 阶段 | 必须产出 | 检查标准 |
|------|----------|----------|
| 市场判断 | MRD | 包含市场分析、用户画像、商业目标 |
| 需求管理 | 需求池+版本规划 | 优先级明确、排期清晰 |
| 需求确认 | PRD+排期表 | 技术可行性确认、设计稿确认 |
| 项目执行 | 可运行版本 | 测试通过、Bug清零 |
| 项目验收 | 上线版本 | 验收签字、推广就绪 |
| 项目上线 | 数据报告+迭代计划 | 效果可衡量、后续有计划 |

### 汇报节点

1. **市场判断后**：向领导汇报方案，预期达成效果
2. **需求确认后**：汇报进度、需要的支持和问题
3. **项目上线后**：汇报结果及迭代计划

## 管理规范（8条硬性要求）

1. **交付物归档**：标红点的内容（评审通过的文档）需在SVN上归档
2. **评审参与原则**：所有评审项，尽量该项目相关人员全员参与；特定环节评审参与人员由PM或项目经理把控
3. **评审方式**：会议或邮件评审，由PM或项目经理把控
4. **评审结论输出**：所有评审项需输出相应的评审结论文档，邮件发相关人员并SVN归档
5. **版本管理**：项目相关输出件，以SVN上版本为主；输出人需确保SVN为最新版；如有改动需邮件通知项目全员
6. **里程碑管控**：里程碑计划表由项目经理制定；里程碑评审确定后，严格按里程碑计划执行；若开发测试过程中有问题/风险，及时反馈给项目经理和PM
7. **上线审批**：项目上线需申请领导审批；上线后由PM邮件通知相关人员（含业务）知悉
8. **任务跟踪**：项目启动后，各阶段任务在禅道上录入进行管理跟踪；需求任务由PM负责，开发测试任务由项目经理负责

## 补充：项目启动会流程

```
确定启动会时间 → 邀请相关人员参与 → 准备启动会内容
                                    ↓
项目背景与价值 → 项目目标 → 项目范围 → 项目计划里程碑 → 项目团队成员与职责
                                    ↓
          会议纪要 → 形成项目团队通讯录 → 明确沟通机制 → 结束
```

| 环节 | 内容 | 负责人 | 说明 |
|------|------|--------|------|
| 项目背景与价值 | 为什么要做这个项目 | PM | 对齐业务目标 |
| 项目目标 | 可量化的成功指标 | PM | SMART原则 |
| 项目范围 | 包含/不包含的内容 | PM | 明确边界防蔓延 |
| 项目计划里程碑 | 关键时间节点 | 项目经理 | 主要阶段deadline |
| 团队成员与职责 | 谁负责什么 | PM | RACI明确 |
| 沟通机制 | 站会频率/周报方式 | 项目经理 | 避免信息断层 |

## 补充：产品方案评审流程

```
评审前准备 → 方案提交 → 评审会（决策/建议/驳回）→ 评审结果确认 → 落地执行
```

| 评审阶段 | 关键动作 | 责任人 | 注意 |
|----------|----------|--------|------|
| **评前** | PM准备评审材料（PRD/原型/流程图），提前1-3天发给参会人 | PM | 确保评审人有时间消化 |
| **评中** | 方案讲解 → 问题讨论 → 决策表决（通过/修改后通过/驳回） | PM主持 | 记录所有问题和决议 |
| **评后** | 输出评审纪要，明确待办和责任人，SVN归档 | PM | 决议必须邮件同步全员 |

**评审结论类型**：
- ✅ 通过：可直接进入开发
- ⚠️ 修改后通过：修改后不需要重新评审
- ❌ 驳回：需要重新设计方案后再次评审

## 补充：产品需求变更流程

```
变更提出方提交变更申请
        ↓
PM评估：变更必要性、影响范围、优先级
        ↓
    ├→ 影响小、优先级高 → PM直接确认，通知相关人
    └→ 影响大、涉及排期 → 提交变更评审 → 领导审批
                                    ↓
                    审批通过 → 更新需求文档/排期 → 通知全员 → SVN归档
```

| 环节 | 关键动作 | 责任人 | 产出物 |
|------|----------|--------|--------|
| 变更提出 | 提交变更需求说明 | 提出方（运营/业务/研发均可） | 变更申请单 |
| PM评估 | 评估必要性、影响范围、技术可行性 | PM + 技术负责人 | 评估结论 |
| 变更评审 | 评审变更方案，决策是否纳入 | PM组织 | 评审决议 |
| 文档更新 | 更新PRD、原型、排期表 | PM | 更新版本文档 |
| 归档通知 | 邮件通知全员，SVN归档 | PM | 变更记录 |

**变更优先级决策矩阵**：
| 级别 | 定义 | 处理方式 |
|------|------|----------|
| P0 紧急 | 不做影响核心功能/数据安全 | 立即排入当前迭代 |
| P1 重要 | 用户体验/业务核心需求 | 评估后排入下一迭代 |
| P2 一般 | 体验优化/锦上添花 | 排入后续版本 |
| P3 低 | 长期规划方向 | 进入需求池待议 |

## 补充：产品迭代规划应用框架

```
版本目标规划（每版本一个核心主题）
        ↓
功能规划 → 将目标拆解为具体功能模块
        ↓
版本优先级 → 三级排序：必做(本次迭代必须完成) / 应做(如有可能尽量完成) / 可做(有余力才做)
        ↓
迭代计划 → 确定版本起止时间、各阶段计划
        ↓
功能验收 → 按优先级逐项验收
```

| 规划环节 | 关键内容 | 方法 | 产出 |
|----------|----------|------|------|
| 版本目标 | 本版本解决什么核心问题 | OKR对齐 | 版本目标声明 |
| 功能拆解 | 目标→功能模块→具体需求 | 用户故事地图 | 功能列表 |
| 优先级排序 | MoSCoW法或价值/成本矩阵 | 多方评估 | 排期计划 |
| 版本命名 | 大版本(3位)/小版本(4位) | 语义化版本 | 版本号 |

**迭代周期建议**：
- 大版本：4-6周一版（含完整设计→开发→测试→验收）
- 小版本：1-2周（补丁/优化/紧急修复）

## 常见问题速查

**Q: 每个阶段大概需要多长时间？**
A: 视项目规模而定，一般：调研1-2周，产研设计2-3周，需求评审1周，开发4-8周，测试2-3周，验收1周，发版1周。

**Q: 需求变更怎么办？**
A: 开发阶段前可接受变更，之后需走需求变更决策流程：评估影响 → PM确认 → 排入后续迭代或紧急变更。

**Q: 如何确保不延期？**
A: 关键：1)排期留buffer 2)每日站会同步进度 3)里程碑评审严格把关 4)风险提前暴露 5)范围严格管控。

**Q: 验收不通过如何处理？**
A: 根据问题严重程度：1)严重问题必须修复才能上线 2)一般问题可排入迭代 3)所有问题需记录并评估影响。

**Q: 测试环境BUG和线上BUG有什么区别？**
A: 测试环境BUG在提测阶段解决；线上BUG需紧急部署修复，可能涉及回滚。线上BUG必须走紧急流程。

**Q: 复盘要做什么？**
A: 复盘关注：1)目标达成情况 2)过程中的问题 3)可复用的经验 4)待改进的点。产出复盘报告并同步全员。
